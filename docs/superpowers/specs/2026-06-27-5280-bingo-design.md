# 5280 Bingo — Design Spec

**Date:** 2026-06-27
**Status:** Approved, ready for implementation planning

## Purpose

A small interactive 2-page website gifted to Rachel (sister) and Steven (brother-in-law)
to welcome them to Colorado. It presents a 4x4 "5280 Bingo" board of 16 real activities
that John and Kelly love, plus a companion map of the Front Range showing where each one
is. Rachel and Steven mark off adventures as they complete them; progress syncs live
between their devices, giving everyone a shared vocabulary of great things to do in
Colorado.

Visual theme: the Denver Broncos 1977 "Orange Crush" throwback palette — royal blue,
white, and orange with retro racing stripes.

## Success Criteria

- Rachel and Steven can open the site on desktop or mobile and see the same shared board.
- Marking a square done on one device updates the other device's board and map in real time.
- The board detects BINGO (any line of 4) and BLACKOUT (all 16) with a celebration.
- The map shows all 16 real locations; finished spots are visually distinct.
- The whole thing runs locally (open a file / one-line static server) and deploys to
  GitHub Pages with no build step.

## Scope

### In scope

- Two pages: Board and Map, sharing one nav, one stylesheet, one data source.
- Shared, real-time, single household board (not per-person) via Firebase Realtime Database.
- BINGO line detection and BLACKOUT celebration.
- Responsive layout for desktop and mobile, including touch (tap) interactions for the map.
- Outbound links to each activity's website.

### Out of scope (YAGNI)

- User accounts / authentication / per-person boards.
- Admin UI for editing activities (activities live in a code data file).
- A build pipeline or framework.
- Visibility/analytics for John (explicitly not needed).

## Pages & Navigation

Two static HTML pages sharing a top nav bar and a single stylesheet:

- **Board** — `index.html` — the 4x4 bingo card. Default landing page.
- **Map** — `map.html` — interactive Front Range map with the same 16 spots as pins.

The nav toggles between the two. Both pages are fully responsive.

## The Board Page

- 4x4 grid; each square represents one activity, shown as a short label.
- Tapping/clicking a square's body toggles its done state.
- **Done state:** square fills orange (`#fb4f14`) with a white X. (Chosen treatment A.)
- **BINGO detection:** any complete row, column, or diagonal of 4 triggers a "BINGO!"
  celebration the first time that specific line is completed (each line celebrates once).
- **BLACKOUT:** when all 16 squares are done, a larger "BLACKOUT!" celebration fires.
- **Outbound links:** each square shows a small link icon (e.g. ↗) that opens that
  activity's URL in a new tab. The icon is a separate tap target so it does not conflict
  with the tap-to-mark gesture on the square body.

## The Map Page

- **Leaflet + OpenStreetMap** base map.
- 16 custom pins, one per activity, placed at the activity's real geocoded coordinates.
- **Popup interaction:** hover on desktop / tap on mobile reveals a popup with the
  activity's description ("Do {activity} at {location}") and a link to its website.
- **Done reflection:** a pin darkens to navy when its corresponding square is marked done,
  so the map and board always reflect the same shared state.
- The view auto-fits to show all 16 spots, which span Denver metro, Boulder, Golden,
  Evergreen, Arvada, and Westminster.

## Shared State & Data Model

- **One shared household board** synced in real time via **Firebase Realtime Database**
  (free tier). A change on one device appears on all others instantly.
- The Firebase web config (including the API key) is embedded in the page source. This is
  acceptable: only Rachel and Steven use the site, on an unlisted URL, and the worst case
  is a stranger toggling squares.
- **Activities data** lives in a single `activities.js` file. Each entry:

  ```js
  { id: "red-rocks", label: "Red Rocks", description: "Hike at Red Rocks Amphitheatre",
    url: "https://www.redrocksonline.com/", lat: 39.6655, lng: -105.2057 }
  ```

  This single source feeds both the board and the map (coordinates are approximate/
  representative and will be refined during implementation).

- **Done state** is stored in Firebase as a set/map of completed activity IDs, e.g.
  `{ "red-rocks": true, "casa-bonita": true }`. Both pages subscribe to this and re-render
  on change. A local fallback (in-memory or localStorage) keeps the UI working if Firebase
  is unreachable, so the site is still usable offline / before config is added.

## Visual Design

- **Palette (Orange Crush '77):** royal blue `#0d52b0` (board), deeper blue `#0b3f8a`
  (squares), orange `#fb4f14` (accent / done fill / X), white `#ffffff`, navy for done
  map pins, gold `#ffd23f` for completed-line glow.
- **Header:** royal blue with an orange-white-orange racing-stripe band.
- **Title:** blocky athletic type ("5280 BINGO") with an orange drop-shadow.
- The same palette and striping carry across both pages so the site feels cohesive.

## Technical Approach

- **Vanilla HTML/CSS/JS, no build step.** Leaflet and Firebase are loaded via CDN
  `<script>` tags. This satisfies "runs locally and on GitHub Pages" with zero tooling.
- **Module split** (each file focused and independently understandable):
  - `activities.js` — the 16-activity data array (single source of truth).
  - `store.js` — Firebase init, subscribe, read/write of done-state; local fallback.
  - `board.js` — render the 4x4 grid, handle toggles, call into store.
  - `map.js` — render the Leaflet map and pins, reflect done-state.
  - `celebrate.js` — BINGO line detection, blackout check, celebration UI.
  - `styles.css` — shared theme.
- **Interfaces:** `store` exposes something like `subscribe(cb)`, `toggle(id)`,
  `isDone(id)`. Both `board` and `map` depend only on `store` and `activities`, not on
  each other.

## Testing

- **Unit tests** for pure functions: BINGO line detection, blackout check, and
  done-state serialization. (Lightweight runner, no framework lock-in.)
- **Manual in-browser verification** for the Firebase sync, Leaflet rendering, and DOM
  interactions, via the run/verify flow.

## Deployment & Running

- New git repo at `/Users/airforce/Documents/GitHub/5280-bingo`.
- **Local:** open `index.html` directly, or run a one-line static server
  (`npx serve` / `python3 -m http.server`) if Firebase localhost behavior needs a real

- **Hosted:** GitHub Pages (serve from `main` or a `gh-pages` branch). No build step.
